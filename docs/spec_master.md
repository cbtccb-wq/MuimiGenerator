---

## 目的

「やたら大げさなのに成果が小さい」機械を自動生成・シミュレーションするWebアプリ。
歯車が回り、レバーが揺れ、リンクが伝達し、最後に旗が少しだけ震える。
**壮大な徒労感の演出**が最終ゴール。

---

## 対象ユーザー

暇を持て余した人、不条理が好きな人、ルーブ・ゴールドバーグ的なものを愛する人。

---

## MVPの到達目標（6条件すべて必須）

1. 新規生成ボタンで機構が1つ表示される
2. 入力部品から出力部品まで、2D上で因果連鎖が見える
3. 再生すると旗またはベルが最終的に反応する
4. 生成結果をJSONで保存・再読込できる
5. 無意味度スコアが表示される
6. 少なくとも3種類以上、見た目と経路が異なる生成結果が出る

---

## 技術スタック

| 技術 | 用途 |
|---|---|
| React + TypeScript | UIフレームワーク |
| Vite | ビルドツール |
| Zustand | 状態管理 |
| SVG | 機構描画 |
| JSON | 保存・読込 |
| Vitest | テスト |

---

## 開発方針

- 初期版は **SVGベース** で構築する
- 物理演算ではなく **tickベース離散シミュレーション** を採用する
- 本格CADではなく **見て納得できる因果連鎖** を優先する
- まず主経路生成を成立させ、その後に冗長経路・変異生成を追加する

---

## 機能一覧

### Must（必須）

- 新規生成ボタン → 機構1件生成
- SVGキャンバスに部品と接続線を表示
- tickベースシミュレーション再生（再生/停止）
- 無意味度スコア表示（0〜100）
- JSON保存・読込
- 部品種別: Handle, Gear, Lever, Cam, Slider, Flag, Bell

### Should（できれば）

- ズーム・パン
- 手動編集（部品追加・移動・削除・接続）
- Undo/Redo
- 冗長経路生成（見た目の豊かさ）
- ステップ実行

### Nice to have（余裕があれば）

- アニメーションの磨き込み
- スコアに応じた自動コメント
- 複数機構の管理

---

## 捨ててよいもの（MVP段階）

- 細かなアニメーション
- 手動編集（生成 → 表示 → 再生の主軸が先）
- Undo/Redo（Phase 8）
- コメント生成の文言磨き込み

---

## 部品モデル

| 種別 | 役割 |
|---|---|
| Handle | 入力起点 |
| Gear | 回転伝達 |
| Lever | 揺動変換 |
| Cam | 回転→往復変換 |
| Slider | 直動部品 |
| Flag | 出力（旗が震える） |
| Bell | 出力（ベルが鳴る） |

接続種別: `gear_mesh` / `push_link` / `trigger` / `cam_follower` / `rope`

---

## 非機能要件

| 項目 | 基準 |
|---|---|
| 操作後の応答速度 | 1秒以内 |
| console error / warning | 0件 |
| lint error | 0件 |
| type error | 0件 |
| APIキーのハードコード | 0件 |

---

## 禁止事項

- 内部IDや変数名を画面に表示しない
- エラーメッセージをそのまま画面に表示しない
- APIキー・秘密情報をコードにハードコードしない
- main ブランチへ直接 push しない
- 物理演算ライブラリを導入しない（tickベース維持）

---

## 合格基準

- 評価採点: 80点以上（`docs/ai/evaluation_rubric.md` 参照）
- 安全性: 20/25点以上（足切り）
- 機能: 35/40点以上
- Vitest: 100% pass
- Level A テスト: 全通

---

## 最低限成果物（Definition of Salvage）

途中停止してもこれが残っていれば「失敗ではない」：

- 固定サンプル機構がSVG描画できる状態
- Handle→Gear→Flag の最小シミュレーションが動く状態

---

## 承認待ち条件

- 本番設定の変更
- 物理演算ライブラリの導入（方針変更に相当）
- 仕様の根本変更（部品モデル・接続モデルの再設計）

---

## 既知リスク

| リスク | 対策 |
|---|---|
| 生成結果がつまらない | 冗長経路・自己干渉・飾り経路を段階追加 |
| シミュレーション破綻 | tickベース単純化・接続種別限定・無限ループ検知 |
| UI実装で時間を食う | 最低限の編集機能に絞る。見た目より操作成立を優先 |
