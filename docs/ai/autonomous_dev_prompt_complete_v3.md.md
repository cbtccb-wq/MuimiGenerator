# 長時間自律開発プロンプト完全版 v2
### VSCode + AIエージェント（Claude Code / Codex / Gemini）対応版

---

## 0. この文書の目的

この文書は、AIエージェントに長時間の自律開発を任せるときに、品質・再現性・安全性・回収可能性を最大化するための完全版ガイドである。  
対象は、個人開発、社内ツール、業務支援アプリ、UI改善、設計支援ツール、PoC、比較的中小規模のWebアプリや自動化ツールを主とする。  
目的は「AIに勝手に頑張らせること」ではない。目的は、**AIが忘れる・盛る・飛ぶ・見た目に逃げる・途中で壊す**という前提のまま、寝ている間でも成果物と証跡が残るように運用を設計することである。

---

## 1. 基本思想

長時間自律開発で重要なのは、モデルの賢さだけではない。以下の5要素を固定することが重要である。

- **檻**: 制約、変更範囲、破壊禁止領域、承認待ち条件
- **餌**: 仕様、優先順位、フェーズ予算、ゴール、最低限成果物
- **足跡**: 差分ログ、証跡、提出パック、仮定ログ、再開手順
- **非常停止ボタン**: 停止条件、危険操作、介入ポイント、承認待ち
- **物差し**: 固定された評価尺度、採点表、合格閾値、固定評価プロンプト

AIに長時間自律作業をさせるとは、天才を呼ぶことではない。  
**多少雑な作業者でも、工程表・テスト・評価表・証跡の中に閉じ込めれば規格品に近づく**ようにすることである。

---

## 2. 適用範囲

この文書は特に以下に向く。

- UIを含むWebアプリ開発
- 社内ツール、管理画面、ダッシュボード
- データ表示・検索・編集系アプリ
- 状態遷移や業務フローを持つアプリ
- 中規模までの自動化、設計支援、PoC

以下では強い調整が必要である。

- 医療、金融、法務、認証、権限、本番課金、削除を含む高リスクシステム
- リアルタイム制御や安全機能を含むシステム
- 法的要件や監査要件が厳しいプロダクト

高リスク用途では、この文書をそのまま使わず、人間レビュー密度と承認ポイントを増やすこと。

## 3. 問題の構造

長時間自律開発で起きる問題は、主に次の層に分かれる。

### 3.1 仕様の問題
- 指示が曖昧で、反証不能
- 「センスよく」「いい感じに」など採点不能な表現が多い
- 優先順位が書かれておらず、全部を同時に大事にしようとする

### 3.2 検証の問題
- 動いたら即コミット
- 自己チェックがなく、壊れたまま進む
- テストの重さ設計がなく、毎回全部回して破綻するか、逆に軽すぎて壊れる

### 3.3 フィードバックの問題
- 人間がスクショや挙動を見て都度修正指示を出す前提になっている
- AIが自分で評価しているつもりでも、評価基準がループ中にドリフトする

### 3.4 記憶の問題
- 長時間回すとコンテキストが劣化する
- handoff を何度も要約して情報が蒸発する
- 次に何をすればよいか分からなくなる

### 3.5 運用の問題
- 予算や時間を序盤に食い潰す
- 曖昧な点で止まりすぎる
- 危険操作と軽微な調整の扱いが同じ
- 修正範囲が爆発して炎上する

## 5. 再現性の固定

改善ループを成立させる前提は、比較可能性である。比較可能性を成立させる前提は、再現性である。

### 5.1 固定すべきもの
- 使用モデルの役割
- モデルごとの温度、max tokens、top_p 等
- 依存関係のバージョン
- ブラウザバージョン
- 実行環境
- seed
- テストデータ
- モックレスポンス
- スクショ撮影条件

### 5.2 必須ルール
- lock file を必須にする
- ランダム処理は seed を固定する
- golden dataset を用意する
- スクショの解像度、ビューポート、テーマ、言語、タイムゾーンを固定する
- 主要APIは必要に応じてモック化し、比較対象を安定させる

### 5.3 推奨ファイル
- `requirements.txt` / `poetry.lock` / `package-lock.json` / `pnpm-lock.yaml`
- `./tests/golden_data/`
- `./config/eval_env.md`
- `./config/model_roles.yaml`

## 6. 単一仕様ソース

仕様が README、Issue、会話ログ、別md、コミットメッセージに散ると、AIは古い仕様を拾って正しい顔をして実装する。  
そのため、仕様の一次ソースを一つに固定する。

### 6.1 単一仕様ソースに含めるべき内容
- 目的
- 対象ユーザー
- 機能一覧
- 非機能要件
- 禁止事項
- 優先順位
- 合格基準
- 最低限成果物
- 承認待ち条件
- 既知の制約

### 6.2 必須ルール
- 仕様の真実は `./docs/spec_master.md` のみとする
- 他文書はこのファイルを参照するだけにする
- 仕様変更時は `spec_master.md` を先に更新する
- 実装は仕様を引用するが、仕様を別所で再定義しない

### 6.3 禁止
- README に別の仕様を書く
- issue コメントにだけ仕様変更を書く
- handoff に仕様を書き換える
- change log を仕様の代わりに使う

---

## 7. 優先順位と捨てるもの

全部大事と言うと、AIは全部を中途半端に大事にする。  
そのため、優先順位と捨ててよいものを明示する。

### 7.1 優先順位の書き方

#### Must
- 主要機能が動く
- クラッシュしない
- 禁止事項を破らない
- 内部情報を漏らさない

#### Should
- 参照サイトのトーンに近い
- ラベルが自然
- 主要ユースケースの操作が分かりやすい

#### Nice to have
- アニメーション
- 微細な装飾
- 細かな余白の磨き込み

### 7.2 捨ててよいもの
- 細かなアニメーションは後回しでよい
- 低優先のサブ機能はダミーでもよい
- 文言の完璧な磨き込みは最終フェーズのみ
- 非本質の装飾は、機能・安全性・可読性の後でよい

---

## 8. 合格基準の数値化

「よい感じ」「見やすく」では、AIが自分を採点できない。合格基準は数値化する。

### 8.1 UI品質の例
- カード内の原文表示は最大5行
- 同時表示ボタン数は最大4個
- フォントサイズ種は3種類以内
- 内部ID / 変数名の表示は0件
- 操作後の画面更新は1秒以内

### 8.2 コード品質の例
- lint error: 0件
- type error: 0件
- unit test: 100% pass
- coverage: 80%以上
- console error / warning: 0件
- APIキーや秘密情報のハードコード: 0件

### 8.3 振る舞い品質の例
- DB空状態でもクラッシュしない
- 0 / 50 / 100 の入力で正しく動く
- 主要操作を20回連続で実行して破綻しない
- 画面縦長・横長でも致命的崩れなし

## 9. タスクグラフとフェーズ設計

AIは優秀な作業者にはなれても、雑な現場監督になりがちである。  
そのため、実装前にタスクグラフを作らせる。

### 9.1 実装前に作るもの
- タスク一覧
- 依存関係
- 高リスク箇所
- 入力 / 出力
- 完了条件
- 検証方法

### 9.2 フェーズ分割ルール
- 1フェーズは 30分〜2時間程度で終わる粒度を目安にする
- 大改修は別フェーズに分離する
- UI、データ、状態管理、エラーハンドリングを分けて扱う

### 9.3 高リスク箇所のみ複数案探索
複数案を競わせるのは有効だが、全体に適用すると重すぎる。  
そのため、**高リスク箇所だけ**初手で枝分かれ探索を行う。

対象例:
- 状態管理方式
- DBスキーマ
- 画面骨格
- エラーハンドリング方針
- 認証フロー

やり方:
- 2〜3案出す
- それぞれの利点・欠点・採用条件を書く
- 同一採点表で比較する
- 採用理由を1段落で残す

## 10. フェーズ別予算と予備費

総予算だけ決めても、序盤で食い潰されることがある。そのため、予算はフェーズ配分する。

### 10.1 管理対象（4系統は独立して管理する）
- トークン予算
- APIコスト上限
- 壁時計時間
- 最大ループ回数

> 金だけ足りても時間で死ぬし、時間だけあっても推論コストで死ぬ。全部別系統の資源として扱うこと。

### 10.2 配分例
- Phase 1-3: 45%
- Phase 4-5: 25%
- 改善ループ: 20%
- 緊急修正バッファ: 10%

### 10.3 ルール
- 各フェーズは開始前に予算を宣言する
- 予算超過時はフェーズを切り上げて handoff を残す
- 緊急修正バッファは通常フェーズで使わない

---

## 11. 禁止事項と破壊禁止領域

ポジティブな仕様だけでは足りない。やってほしくないことを、同じだけ明示する。

### 11.1 禁止事項例
- DBの内部IDを画面に表示しない
- 内部変数名を表示しない
- 原文欄に要約や説明を混ぜない
- エラーメッセージをそのまま画面に表示しない
- 同一著者の原文を関連引用として再利用しない

### 11.2 破壊禁止領域例
- migrations を勝手に削除しない
- auth / secrets / billing 周辺は承認なし変更禁止
- 既存テストを消さない
- main ブランチへ直接 push しない
- 本番設定を更新しない
- データ削除操作を自動実行しない

---

## 12. テスト戦略の階層化

毎ループ全テストは重い。逆に軽いテストだけでは壊れたまま進む。  
そのため、テストは3層に分ける。

### 12.1 Level A: 毎ループ必須
- lint
- type check
- smoke test
- 主要画面の起動確認
- console error / warning 確認

### 12.2 Level B: フェーズ完了時
- unit test 全体
- 主要ユースケース
- 軽量境界テスト
- visual regression
- 主要API / 主要状態遷移テスト

### 12.3 Level C: 最終承認前
- full regression
- 長時間操作テスト
- 負荷テスト
- 既知バグ再発確認
- 最終スクショ比較

### 12.4 軽量境界テスト（全プロジェクト必須）
ファズテストを重く考えすぎない。最低限、以下は全プロジェクトで確認する。

- 空文字
- null
- データ0件
- 超長文
- 特殊文字
- 同一操作連打
- API遅延
- 切り替え連打
- 画面縦長 / 横長

---

## 13. バグ修正ルール: RED→GREEN 証跡

「修正前に失敗するテストを書け」だけでは不十分である。AIは修正後に通るテストを書いて、それを再現テストだと主張することがある。

### 13.1 必須手順
1. 再現テストを書く
2. 修正前コードでそのテストを実行する
3. **RED であることをログに残す**（この記録がないと無効）
4. 修正する
5. 修正後に再実行する
6. **GREEN であることをログに残す**
7. 回帰テスト群に追加する

### 13.2 保存物
- REDログ
- GREENログ
- 変更対象ファイル
- 関連コミットハッシュ

## 14. 自己評価ループ

### 14.1 各フェーズ完了時に行うこと
1. アプリを起動する
2. Playwright 等でスクショを撮る（条件は固定する）
3. 固定された評価プロンプトで採点する
4. 問題があれば修正する
5. 再スクショ・再採点する
6. 上限回数に達するか、合格閾値に達したら終了する

### 14.2 自己評価ループの上限
- 1フェーズあたり最大3回を目安
- 改善率が一定以下なら打ち切る
- 同一指摘が続くなら停止して記録する

## 15. 評価尺度の固定（評価者のドリフト防止）

改善ループの中で評価者まで変動すると、計測器が壊れる。  
同じモデルに「作らせて」「採点させて」を繰り返すと、ループが進むにつれて採点が甘くなり自作自演の審査会になる。  
したがって、評価基準はループの外で固定し、ループ中は書き換えさせない。

### 15.1 固定するもの
- 採点プロンプト
- 採点項目
- 項目順序
- 配点
- 合格閾値
- 足切り条件
- 比較対象

### 15.2 必須ルール
- 評価プロンプトは `./config/eval_prompt_v1.md` に保存し、**ループ中は変更禁止**
- 変更する場合は新バージョンとして扱う（eval_prompt_v2.md）
- ループ中に採点軸を書き換えない
- 評価時のビューポートやテーマも固定する

### 15.3 比較対象の固定
- 初期基準スクショ
- 参照サイトスクショ
- 直前フェーズスクショ

必要に応じて、節目だけ別評価器で spot check する。

## 16. 重み付き採点と判定ルール

単純なチェックリストでは、全部1点扱いになる。  
しかし実務では、クラッシュしないことは余白の美しさより遥かに重要である。

### 16.1 重み例
- 機能: 40
- 安全性: 25
- 可読性: 15
- 性能: 10
- 見た目: 10

### 16.2 判定ルール例
- 総合80点以上で合格
- 安全性 20/25 未満なら総合点に関係なく不合格
- 機能 35/40 未満なら改善ループ継続
- 見た目項目は最大2ループまでしか追わない

### 16.3 重要な考え方
- 重みだけでなく、**足切り条件**も固定する
- UIの小改良が機能や安全性を押しのけて主目標化しないようにする

## 17. モデル役割分離

モデル名をベタ書きすると、テンプレの寿命が短い。そのため、モデル名ではなく**役割階層**で書く。

### 17.1 役割例
- 低コスト視覚モデル: 日常的なUI確認、軽いバッチ処理
- 中コスト高精度モデル: コードレビュー、主要判定
- 高コスト推論モデル: 最終判定、難所の比較検討（節目のみ）

### 17.2 ルール
- 高価な評価器は毎ループ使わない
- 節目だけ高精度評価を行う
- 実装モデルと評価モデルは可能な限り役割分離する
- UI評価とコード評価は可能なら別役割にする

---

## 18. 仕様曖昧エラーの扱い

仕様曖昧は止めすぎると、自律開発が進まない。  
そのため、原則は **記録して進む** とし、危険領域だけ停止対象にする。

### 18.1 基本ルール
- 仕様曖昧が発生したら暫定判断で進む
- `assumptions.md` に記録する
- `human_review_flags.md` に確認フラグを立てる
- 危険領域に触れる場合のみ承認待ちまたは停止する

### 18.2 曖昧さの分類

| 種別 | 例 | 対応 |
|---|---|---|
| 軽微曖昧 | UI文言・並び順・余白 | 暫定判断で進む |
| 中程度曖昧 | 状態遷移の一部・非本質例外 | 暫定＋強い確認フラグ |
| 重大曖昧 | 認証・課金・削除・権限・本番影響 | 承認待ちまたは停止 |

---

## 19. 変更爆発半径の制限

AIは一箇所の修正のつもりで、関連しそうな場所を連鎖的に触って大火事にすることがある。  
そのため、1ループごとの変更範囲を制限する。

### 19.1 ルール例
- 1ループで変更してよいファイルは最大5個
- 200行超の変更は大規模変更として別ブランチへ分離
- migrations / auth / secrets 周辺は承認なし変更禁止
- リファクタは専用フェーズに分離

---

## 20. 停止条件

停止条件がないと、AIはどうでもよい細部を磨き続けるか、同じ失敗を繰り返す。

### 20.1 停止条件例
- 改善スコアが3ループ連続で1%未満
- 同一エラーが3回連続
- フェーズ予算超過
- 最大実行時間超過
- 最大ループ回数到達
- 危険操作に到達して承認待ち

### 20.2 エラー分類別対応

| 種別 | 例 | 対応 |
|---|---|---|
| 一時的失敗 | ネットワーク・APIタイムアウト | 短回数リトライ |
| 決定的失敗 | 同じテストが何度も落ちる | ログ化して戦略切り替え |
| 仕様曖昧失敗 | 正解不明 | 暫定判断で進む＋assumptions＋review flag |
| 危険操作失敗 | DB削除・本番設定変更 | 停止 / 承認待ち |

## 21. 3層介入設計

全部自律にすると危険だが、何でも確認待ちにすると進まない。  
そのため、介入ポイントを3層に分ける。

### 21.1 レベル1: 完全自動でよい
- UI微修正
- 文言調整
- テスト追加
- 軽いリファクタ

### 21.2 レベル2: 通知のみで続行
- 新規依存追加
- 設定変更
- 中程度構造変更
- コスト増が見込まれる処理

### 21.3 レベル3: 必ず停止して承認待ち
- DB削除 / 破壊的変更
- 本番設定変更
- 権限変更
- 秘密情報変更
- main 直 push
- 外部課金操作
- 認証・課金・削除の重大仕様変更

## 22. handoff と resume

handoff と resume は似ているが違う。

- **handoff**: ここまで何をやったか、何が残っているか
- **resume**: 次に何を打てばよいか、どこから再開すべきか

### 22.1 handoff に入れるもの
- このフェーズで変えたこと
- 変えなかったこと
- 理由
- 未解決課題
- 現在の仮定
- 触ってはいけないもの

### 22.2 resume に入れるもの
- 現在フェーズ
- 直前成功コマンド
- 次に打つべきコマンド
- 失敗している箇所
- スキップしてよい箇所
- 再開時の確認ポイント

## 23. サマリー劣化防止: 差分追記 + 節目スナップショット

handoff を何度も要約し直すと、情報が蒸発する。  
そのため、日々の記録は**差分追記**とし、節目だけ全体スナップショットを取る。

### 23.1 差分追記で毎回残すもの
- 今フェーズで変えたこと
- 変えなかったこと
- その理由

### 23.2 節目スナップショット
- 3〜5フェーズごとに全体状態をスナップショット化
- 版番号付きで保存（context_snapshot_v1.md, v2.md...）
- 上書きではなく追記管理する

## 27. 最低限成果物 (Definition of Salvage)

完走を前提にすると、途中で壊れた時に何も残らない。

### 27.2 例

## 31. VSCode + AIエージェント運用ガイド

この文書をVSCodeでClaude Code / Codex / Gemini Code Assistに使わせるための実践手順。

### 31.1 基本方針：母艦→短縮版の二段構え

この完全版mdを毎回全文貼るのはコンテキスト圧迫で崩れる。  
正しい使い方は：

1. この完全版を `docs/ai/長時間自律開発プロンプト完全版.md` に置く（読むだけの原典）
2. AIに「このプロジェクト専用の短縮ルール群」を生成させる（実行用）

### 31.2 リポジトリ構成

```
your-project/
├─ docs/
│  └─ ai/
│     ├─ 長時間自律開発プロンプト完全版.md  ← この文書
│     ├─ agent_operating_rules.md          ← 短縮版（AIに生成させる）
│     ├─ evaluation_rubric.md              ← 採点表（ループ中変更禁止）
│     ├─ task_template.md
│     ├─ context_handoff.md
│     ├─ resume.md
│     ├─ assumptions.md
│     └─ phase_report_template.md
├─ CLAUDE.md                               ← Claude Code用（必須事項のみ）
├─ .claude/
│  └─ rules/
├─ .codex/
│  └─ config.toml                          ← Codex用設定
└─ src/
```

### 31.3 最初の1回だけやること（初期化プロンプト）

```
docs/ai/長時間自律開発プロンプト完全版.md を読み、
このリポジトリ向けに運用可能な短縮ルール群へ分解してください。

作成対象:
- docs/ai/agent_operating_rules.md
- docs/ai/evaluation_rubric.md  ← 作成後はループ中変更禁止と明記すること
- docs/ai/task_template.md
- docs/ai/context_handoff.md の初期フォーマット
- docs/ai/resume.md の初期フォーマット
- docs/ai/assumptions.md の初期フォーマット

条件:
- このプロジェクトに不要な一般論は削る
- 危険操作・停止条件・評価尺度・テスト階層・爆発半径制限を必ず残す
- evaluation_rubric.md は「このファイルはループ中変更禁止」と冒頭に明記する
- 最後に「このプロジェクトで最重要な10ルール」を箇条書きで出す
- まだコードは変更しない
```

### 31.4 エージェント別の使い方

#### Claude Code

CLAUDE.md は **大きくなりすぎると読み飛ばされる**（数百行以内推奨）。  
重要事項だけを入れ、詳細は `.claude/rules/` に分割する。

CLAUDE.md に入れるべき7項目：
- 単一仕様ソースの場所
- テスト階層 A/B/C の定義
- 破壊禁止領域リスト
- 変更爆発半径制限（最大5ファイル）
- handoff と resume を更新する条件
- evaluation_rubric.md の場所（変更禁止と明記）
- 危険操作は承認待ちにすること

毎回の依頼プロンプト：
```
まず以下を読んでください:
- docs/ai/agent_operating_rules.md
- docs/ai/evaluation_rubric.md
- docs/ai/resume.md
- docs/ai/assumptions.md

次に、今回の作業タスクを3〜7ステップに分解し、
各ステップごとに
1. 変更対象ファイル
2. 実行する検証（Level A/B/C）
3. 停止条件
を提示してください。

その後、Step 1 から実装してください。
危険操作は停止して報告、
仕様曖昧は assumptions.md に記録して暫定判断で進んでください。
```

#### Codex（VSCode拡張）

`.codex/config.toml` に動作設定、`docs/ai/*.md` に運用規約を分ける。

毎回の依頼プロンプト：
```
Read these files first and treat them as the operating policy for this repo:
- docs/ai/agent_operating_rules.md
- docs/ai/evaluation_rubric.md
- docs/ai/resume.md
- docs/ai/assumptions.md

Before changing code:
1. summarize the current task in 5 lines
2. propose a plan with test level (A/B/C) for each step
3. identify risky operations
4. state which files you will modify (max 5 files per loop)

Then implement only Step 1.
Keep changes within the blast-radius limits defined in the rules.
```

#### Gemini Code Assist

```
Read and follow these repository rules before coding:
- docs/ai/agent_operating_rules.md
- docs/ai/evaluation_rubric.md
- docs/ai/resume.md

Task:
[ここに今回の開発タスク]

Requirements:
- propose phased plan first
- use test levels A/B/C
- keep a context handoff entry after each phase
- record assumptions instead of silently guessing
- do not modify protected areas
- max 5 files changed per loop
```

### 31.5 毎回の依頼は3点セットで出す

どのエージェントでも、毎回の依頼をこの形で出すことで崩れにくくなる。

```
【今回のタスク】
（何を作るか）

【必ず読む】
- docs/ai/agent_operating_rules.md
- docs/ai/evaluation_rubric.md
- docs/ai/resume.md
- docs/specs/（該当仕様）

【完了条件】
- （機能条件）
- Level A テスト全通
- Phase完了時に context_handoff.md 更新
- assumptions.md に未確定事項を記録
- 変更ファイルは最大5つまで
```

### 31.6 やってはいけない使い方

- 毎回、完全版mdを丸貼りする（コンテキスト圧迫で崩れる）
- ルールを1ファイルに全部押し込む（resume と rubric と assumptions が埋もれる）
- 1回の依頼で大改修までやらせる（爆発半径が大きくなる）
- handoff を要約だけにする（差分追記＋節目スナップショットが正しい）

### 31.7 毎日の推奨フロー

1. resume.md を読む
2. その日のタスクを1つに絞る
3. エージェントに「読むべきファイル」を指定して plan を出させる
4. Step 1 だけ実装させる
5. Level A テストだけ通す
6. 問題なければ次の step へ
7. Phase が終わったら context_handoff.md と assumptions.md を更新
8. 節目で phase_report.md と固定証跡を残す

## 33. 運用上の注意

- この文書は、AIを信じるためのものではない。AIを**観測可能な実験系**に閉じ込めるためのものである。
- 仕様曖昧で止まりすぎる運用は、長時間自律と相性が悪い。危険領域以外は、記録して進むこと。
- UIの見た目改善は、機能・安全性・回収可能性を満たした後に行うこと。
- ログは多いほどよいわけではない。差分追記と節目スナップショットで、情報の腐敗と掘削コストの両方を抑えること。
- 複数案競争は全体ではなく、高リスク部分に限定すること。
- 評価者のドリフトを避けるため、評価尺度は固定し、ループ中に改変しないこと。
- CLAUDE.md は簡潔に保つこと。大きすぎると読み飛ばされる。
- 完全版mdは原典として保存し、毎回のプロンプトには短縮版を使うこと。

## 34. 最終要約

長時間自律開発に必要なのは、次の5つである。

- **檻**: 制約・禁止領域・爆発半径制限
- **餌**: 仕様・優先順位・予算・最低限成果物
- **足跡**: handoff・resume・差分ログ・証跡・提出パック
- **非常停止ボタン**: 停止条件・介入設計・危険操作承認待ち
- **物差し**: 固定評価プロンプト・採点表・閾値・足切り条件

AIに長時間自律作業をさせるとは、賢い存在に願うことではない。  
**忘れる、飛ぶ、盛る、雑に触る存在でも、翌朝に成果物と再開手順と証跡が残るように仕組みを組むこと**である。

## 35. マルチエージェントオーケストレーション

複数のAIエージェントを役割分担させて、自動で最後まで走らせる設計。

### 35.1 基本思想

エージェントは役割に固定しない。大事なのは**インターフェースの標準化**。

```
どのエージェントが担当しても
「同じ入力を受け取って」
「同じ形式で出力する」
ように設計すれば入れ替え自由
```

### 35.2 標準化するインターフェース

```
input/
├── spec_master.md        ← 仕様（誰が読んでも同じ）
├── evaluation_rubric.md  ← 採点基準（誰が採点しても同じ）
├── resume.md             ← 再開手順（誰が引き継いでも同じ）
└── task.md               ← 今回のタスク

output/
├── code/                 ← 実装結果
├── test_results.txt      ← テスト結果
├── screenshot.png        ← 画面確認
├── git_diff.txt          ← 変更差分
├── changed_files.txt     ← 変更ファイル一覧
├── review_result.md      ← レビュー結果（合格/差し戻しリスト）
└── phase_report.md       ← 何をやったか
```

### 35.3 エージェント別の得意不得意（現状）

| エージェント | 得意 | 向いてる役割 |
|---|---|---|
| Claude | 長文読解・仕様定義・レビュー・日本語 | 仕様作成・クロスチェック |
| Codex | コーディング・ファイル操作・コマンド実行 | 実装・テスト実行 |
| Gemini | コード変換・補完・検索連携 | リファクタ・ドキュメント生成 |
| QWEN | コスト安・多言語 | 軽いタスク・翻訳系 |
| DeepSeek | コード特化・コスト安 | 実装・コードレビュー |

これは現時点の傾向であり、モデルの進化で変わる。役割はモデル名ではなく得意分野で選ぶこと。

### 35.4 パイプラインの設計

```
人間：要件を一言書く（5分）
  ↓
Claude：spec_master.md を生成
  ↓
Codex：実装 + テスト実行
  ↓
Playwright：スクショ自動撮影
  ↓
Claude：仕様整合性レビュー    ← 同時並行でも可
DeepSeek：コード品質レビュー  ←
  ↓
両方合格 → 次フェーズへ
不合格 → Codexに差し戻し（最大3回）
  ↓
全フェーズ完了 → 提出パック生成
  ↓
人間：15分確認 → 承認
```

### 35.5 Claudeがクロスチェックで見る内容

各Phase完了時に以下をClaudeに投げる：

```
spec_master.md と evaluation_rubric.md を読んだ上で、
以下をクロスチェックしてください：
- git_diff.txt（コード変更内容）
- test_results.txt（テスト結果）
- screenshot.png（画面状態）

チェック観点：
1. 仕様との整合性（spec_masterに書いてない機能が生えていないか）
2. コード品質（型安全・エラー処理・重複・命名一貫性）
3. セキュリティ（APIキー露出・入力値の未検証）
4. テスト網羅性（重要ケースが抜けていないか）
5. UI品質（スクショとspecの差分）

結果を以下の形式で出力してください：
- 合格：次のPhaseへ進んでよい
- 不合格：[具体的な修正箇所・優先度・理由]
```

### 35.6 コードレビューで確認する項目

| チェック項目 | 具体例 |
|---|---|
| 型安全性 | anyの乱用・型アサーションの危険な使い方 |
| エラー処理 | try-catchがない・エラーが握り潰されてる |
| 仕様との差分 | spec_masterに書いてない機能が生えてる |
| セキュリティ | APIキーのハードコード・入力値の未検証 |
| 重複コード | 同じロジックが3箇所以上ある |
| 命名の一貫性 | spec_masterの用語と変数名がズレてる |

### 35.7 エージェント間引き継ぎプロトコル

```
【ルール】
- input/ と output/ のフォーマットを全エージェント共通にする
- エージェントを切り替える時は resume.md を必ず読ませる
- 前エージェントの output/ を次エージェントの input/ にする
- review_result.md の形式を統一する
  （合格/不合格・不合格箇所・修正優先度）

【エージェント選択基準】
- コスト重視 → DeepSeek / QWEN
- 品質重視 → Claude / GPT-4o
- 実装速度重視 → Codex / Claude Code
- 日本語重視 → Claude
- コード変換重視 → Gemini
```

### 35.8 人間確認の定義

```
【確認時間の目標】15分以内
【確認方法】output/final_package/human_review_checklist.md のみ
【確認範囲】変更したPhaseのスクショ差分のみ
【承認方法】「OK」と次のエージェントに返信するだけ
【介入条件】Level 3（DB削除・本番設定変更・権限変更）のみ
```

### 35.9 human_review_checklist.md の自動生成

各Phase完了時にエージェントが自動生成する。内容は：

```
# Phase XX 確認チェックリスト（所要時間：約5分）

## 確認してほしいこと（これだけでOK）
- [ ] screenshot.png を開いて画面が崩れていないか目視確認
- [ ] 変更した機能を1回だけクリックして動作確認
- [ ] review_result.md の「合格」を確認

## 承認方法
問題なければ「OK」と返信してください。
問題があれば該当箇所を一言教えてください。
```
